InsWorks-GDI 命名规范

v1.0 2025-04-26

本文档适用于 InsWorks-GDI 软件及 SDK 的开发的程序命名规范。

  1. 开发环境

    VisualStudio : Visual Studio 2019 (v142)

    QT : 5.12.12

    Ubuntu : 22.04 LTS

    spdlog : 日志库

    nlohmann : json 库 version 3.11.3

  2. 软件通用命名说明

    本段介绍所开发软件的命名及含义解释，以及各个产品的命名，后续各个模块命名，包括文件夹、文件、命名空间应采用以下的产品统一命名方式进行命名。

    • 软件命名：InsWorks-GDI，其中 InsWorks 为苏映视软件家族，GDI 是 General Device Interface 的缩写，即通用设备接口，用于对成像硬件提供统一的用户接口及软件操作习惯。
    • 微距相机命名：LineX2 为微距相机 2 代产品
    • 线扫相位偏折系统：PMDL 为 Phase measuring deflectometry LineScan 缩写
    • 面阵相位偏折系统：PMDA 为 Phase measuring deflectometry AreaScan 缩写
    • 线扫光度立体系统：PMSL 为 Photometric stereo LineScan 缩写
    • 面阵光度立体系统：PMSA 为 Photometric stereo AreaScan 缩写

  3. 文件夹的命名

    对于程序中的文件夹命名进行定义。

    • 最外层文件夹命名为大驼峰加中横线的方式：InsWorks-GDI
    • 其他内层文件夹，包括模块名，模块内部的头文件源文件夹等统一采用小写+下划线
    • 各模块中应将头文件和源文件单独存放，第三方库文件应统一管理存放，切勿混杂在一起，第三方库加入版本信息
    • 文件夹命名应简单清晰明了，且同类型的模块应具有相同的前缀，如驱动类的文件夹，应统一以 driver_开头
    • 相同模块放在统一文件夹，例如驱动相关的放在 driver 文件夹

  5. 头文件主要内容

    头文件应具备防卫式声明，这里考虑兼容性、扩平台，仍采用宏式声明，且需要与文件名称保持一致，必要时考虑增加软件及模块名以确保唯一性，全大写，加 _ 作为前缀，以 \_H_作为后缀，暂不采用#pragma once 这种现代 C++风格。

  6. 命名空间 namespace

    为防止类名冲突，应增加相应的命名空间以加以区分，其中 SDK 部分，统一采用 gdi 作为命名空间，与微距相机相关的以 linex2。

  7. 类名称说明

    类定义中应将私有属性、方法与公开方法做好区分，切勿将属性直接向外暴露，应提供对应的 set/get 方法。且类声明中应包含默认构造、拷贝构造、拷贝赋值以及析构函数，且管理好类中的指针与内存。

    • 类名采用大驼峰命名法：如 FrameBuffer
    • 抽象类或者接口类，增加前缀 I ：如 IBuffer
    • 私有属性采用小写+下划线连接，且在尾部增加额外下划线：如 size_
    • 属性采用小驼峰命名方式：如 void setSize(int size)

  8. 关于结构体的命名

    结构体名应以全大写加下划线方式命名，内部以小写加下划线命名，对外接口的结构体应保持 C 风格，不应包含 C++内容。

  9. 关于枚举类型的命名

    • 枚举命名全大写加下划线，需要为属性设置初始值。
    • 枚举定义为 enum 不允许使用 enum class
    • 必须使用 typedef
    • 枚举类型的命名为：INS_PREFIX_FUNCTIONAL_NAME 其中，PREFIX 表示前缀，用来区分 FUNCTIONAL 功能类似的枚举。
    • 枚举内容定义为 INS* + PREFIX*(表示前缀) + 功能描述 + 具体枚举内容
    • 必须标识所有枚举的具体值

    为什么是 C 接口？因为 C 接口具有良好的 ABI（Application Binary Interface）稳定性，可以被多种语言调用（如 Python、C#、Java、LabVIEW、MATLAB 等），而 C++ 的 ABI 在不同编译器和编译选项下可能不兼容。

    C 接口函数统一以 INSGDI作为前缀，且函数名主体以大驼峰进行命名。
